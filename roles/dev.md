# Dev：技术设计与实现责任

## Responsible human

指定的开发人员（Dev）对技术设计、业务仓分支、代码提交和本地验证负责；AI 助手不能代替代码所有者批准外部写入。

## Required input

- BA 已确认的 `requirements.md` 与当前来源快照；
- 目标业务仓、当前分支和已有接口约束；
- QA 的测试重点或失败证据（如已提供）。

## Output

- `design.md` 和 `plan.md` 中的跨仓影响、实施顺序与回滚/恢复策略；
- 每个业务仓的分支、commit、MR 和验证证据引用；
- 修复阶段的复现、根因、最小修正和回归计划；
- 对需求歧义、依赖或失败验证的明确暂停记录。

## Handoff

向 QA 交接可部署的变更范围、测试命令、已知风险、证据链接与下一步。任何创建 MR、push 或更新外部事项的 L2 写入都先展示精确预览，并等待真人提供与该预览绑定的确认 token。L3 操作禁止执行。

## Tool interaction

Dev 在 AI 工具中通过 `/saber`、`/saber-status` 或自然语言要求设计、实现、修复和交接。工具在后台读取需求和仓库证据、维护阶段文档与交接记录，并在 Dev 明确给出阶段结论后调用合法的工作流接口；Dev 不需要操作工作项状态机 CLI。
