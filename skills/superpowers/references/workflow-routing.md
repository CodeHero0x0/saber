# Superpowers 工作流选择路由

选择流程的目标是降低不确定性，不是增加仪式。先确认是否已有批准的规格，再按风险和变更类型选择一个主流程；需要时叠加验证或审查流程。

| 信号 | 首选流程 | 产物或退出条件 |
| --- | --- | --- |
| 新功能、架构方向或需求仍有歧义 | brainstorming | 已确认的范围、非目标和方案选择 |
| 已确认的跨文件/跨仓变更 | writing-plans | 可执行步骤、依赖和验证计划 |
| 行为回归或可明确的功能契约 | test-driven-development | 聚焦失败测试与通过证据 |
| 测试失败、错误或行为不符合预期 | systematic-debugging | 可复现根因与最小修复 |
| 重要实现完成，需要独立检查 | requesting-code-review | 规格与质量问题均已关闭 |
| 准备声明完成、提交或合并 | verification-before-completion | 新鲜命令输出和范围复核 |

## 例外与暂停

- 机械性资产编辑可采用比例验证，但不能跳过 YAML/Markdown 链接和构建检查。
- 外部写入、需求漂移、缺少访问权限或无法复现的失败必须暂停并升级给真人负责人。
- 不要用流程名称绕过 L2 预览确认或把 AI 人设当作权限。
