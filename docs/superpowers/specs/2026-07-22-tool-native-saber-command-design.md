# 工具原生 Saber 命令与通用需求来源设计

## 目标

Saber 的命令行退回为管理员和 AI 技能的内部接口：负责初始化、配置、更新团队资产、物化工具运行时内容和本地工作项读写。团队成员的日常研发入口是其已连接的 Codex、Claude Code 或 OpenCode，通过一个 `/saber` 超级命令和少量辅助命令，以自然语言完成 BA、Dev、QA 工作。

需求来源不再限定 Jira。聊天讨论、Jira、项目文档和人工录入都可成为可追溯输入；其内容快照和指纹保存在工作项中。Jira 仍可作为一种来源，而不是流程的前置条件。

## 非目标

- 不把完整聊天记录、凭证或长日志写入工作项。
- 不让角色推断代替真人责任、审批或 L2 外部写入确认。
- 不建设独立 Web 服务、聊天平台代理或新的外部持久化系统。
- 不自动执行当前 `grill-me` 的 `/grilling` 命令，因为其契约明确禁止模型自动调用。

## 命令体验

团队成员只需记住：

```text
/saber <描述或工作项编号>
```

`/saber` 的处理顺序如下：

1. 若用户明确表达角色或动作，例如“以 QA 身份验证”“修复这个缺陷”，优先采用该意图。
2. 若输入包含已有工作项，读取其状态机；当前状态责任角色优先于默认角色。
3. 若没有工作项，使用当前物化角色作为默认上下文，并根据自然语言判断 BA、Dev 或 QA。
4. 若仍不明确，只问一个最小澄清问题，绝不跳过流程。
5. 输出当前阶段需要的中文产出、证据缺口和下一步；所有外部 L2 写入仍先 preview 并等待精确确认 token。

角色推断只是工作流路由。角色档案是上下文，不是授权，不能让 AI 代表 BA 接受需求、代表 Dev 推送代码或代表 QA 伪造测试结论。

辅助命令保持最小集合：

```text
/saber-intake <描述>       # 强制从 BA 需求接入开始
/saber-focus <工作项编号>  # 加载工作项、仓库、证据与当前阶段
/saber-status [工作项编号] # 只读查看进度、缺失产出和建议动作
/saber-refine              # 结合已选文档深化当前需求草稿，必要时引导 /grilling
/saber-help                # 按当前阶段显示可做事项
```

`/saber-intake` 以中文逐项澄清业务目标、范围/非范围、验收标准、风险和受影响仓库。它会先生成草稿并展示给用户确认，确认前不创建工作项。复杂、矛盾或高影响需求会提示用户显式发起 `/grilling`；其结论由 AI 回填到草稿中，并标明来自哪份文档或项目上下文。

## 通用需求来源

工作项元数据以 `source` 取代 Jira 专属的 `jira`：

```yaml
source:
  kind: chat # chat | jira | document | manual
  title: "订单备注长度需求讨论"
  snapshot: intake.md
  fingerprint: sha256:...
  capturedAt: 2026-07-22T12:00:00.000Z
  references:
    - docs/order-notes.md
```

`intake.md` 是经过用户确认的输入快照，包含初始描述、澄清问答摘要、引用文档和未决项，不保存完整聊天历史。Saber 对该文件计算 SHA-256 指纹。`requirements.md` 保持为 BA 澄清后的正式产出，两者不能互相替代。

工作项只允许一套 `source` 模型。原有模板、demo 和测试一次性迁移；包含旧 `jira` 字段的元数据会明确报错，不在运行时映射、回退或自动升级。状态、漂移检测和 BA 阶段门禁统一使用来源名称和来源指纹。对聊天来源，指纹比对的是已确认的本地快照；若需求发生变化，BA 必须创建新的草稿、更新快照和指纹，再继续流转。

聊天来源的编号由 Saber 自动生成：`SABER-YYYYMMDD-NNN`。生成器扫描当日已有工作项并递增序号；最终通过现有的独占目录创建保证并发时不覆盖。Jira 来源可继续使用现有业务键，但也允许使用自动编号。

## 初始化与资产物化

管理员在仓库中执行 CLI 初始化时选择工具和默认角色。例如，`saber use ba --tool codex` 会把以下内容物化到该工具的原生发现目录：

- 始终安装 `/saber` 及五个辅助命令技能。
- 安装当前角色所需的团队角色说明、工作流、模板和推荐技能。
- 安装团队批准的外部技能和已配置的低风险能力。
- 安装一个聚合上下文，说明默认角色仅用于路由，实际角色仍由 `/saber` 的意图和工作项状态推断。

`saber use dev --tool claude --project backend` 或 `saber use qa --tool opencode --project frontend` 使用相同语义，仅投影到各工具的原生发现目录。用户不需要在日常工作中阅读或输入 Saber CLI；已物化的技能可以在工具中调用本地 Saber CLI 来创建、读取和更新本地工作项。

团队配置仍是基线。用户可在不提交的 `saber.local.yaml` 中声明个人附加技能、已批准的 MCP、提示词包和默认工具/角色；物化时合并这些资产。个人配置不能新增团队项目、改变项目路径、提升风险等级、绕过确认，或替换团队定义的核心 `/saber` 命令。

## 工作流与落盘

1. 用户在工具中输入 `/saber` 或 `/saber-intake` 加描述。
2. AI 读取当前角色上下文、相关工作项和用户指定的项目/文档，开始中文澄清。
3. AI 显示中文需求草稿；用户确认后，由技能在后台调用内部 CLI 创建工作项。
4. 对无 Jira 来源，生成 `SABER-YYYYMMDD-NNN`，写入 `intake.md`、`requirements.md` 模板、`source` 元数据和来源指纹。
5. 工作项从 `ba-clarify` 开始，沿用 BA → Dev → QA → Dev 修复/QA 回归 → BA 验收状态机。
6. `/saber` 在每次后续输入中根据工作项状态进入该阶段，并将阶段性中文产出记录到既有设计、计划、测试、验收和交接文件。

## 验证范围

- `source.kind` 的 chat、jira、document、manual 校验，并拒绝包含旧 `jira` 字段的元数据。
- 自动生成的 `SABER-YYYYMMDD-NNN` 编号、同日递增、冲突重试和不覆盖已有工作项。
- 聊天来源将确认后的 `intake.md` 快照与 SHA-256 指纹写入工作项，且不接受命令行短文本。
- 来源指纹变更会阻断 BA 交接和验收，不影响既有 Jira 行为。
- 各工具物化通用命令、当前角色推荐技能和个人低风险扩展；角色切换不会删除个人资产。
- `/saber` 路由规则的单元测试：显式意图、状态机、默认角色、歧义追问和角色冲突。
- 中文 demo 从聊天需求草稿开始，覆盖 BA、Dev、QA、一次修复闭环和 BA 验收。
