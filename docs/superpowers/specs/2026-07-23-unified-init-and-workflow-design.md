# OpenSpec 变更提案：统一初始化与对话驱动工作流

- 状态：已批准
- 工作项：当前 Saber 脚手架减法
- 提案负责人：Saber 仓库维护者
- 最后更新：2026-07-23

## 问题与目标

当前实现要求管理员先选择 BA、Dev 或 QA 角色物化上下文，并在阶段切换时生成严格的角色交接文件。实际团队成员往往会在同一对话中澄清需求、实现、测试和修复；预先绑定人的角色与强制交接增加了操作和产物，但没有增加可复查证据。

目标行为是：团队成员克隆 Saber 后，只选择自己的 AI 工具执行一次 `saber init`，获得统一 `/saber` 能力。之后由对话意图和工作项状态动态进入需求、开发、测试或修复阶段。

## 范围与非目标

- 范围：统一初始化入口、无角色物化、统一 `/saber` 路由、取消强制 handoff、MCP 原生配置安装与卸载。
- 范围：直接删除旧 CLI、旧 schema、旧角色物化实现和现有历史工作项，不提供兼容层或迁移工具。
- 范围：保留来源指纹、工作项状态、阶段门禁、仓库引用、测试证据和最终 BA 接受。
- 非目标：取消 BA/Dev/QA 阶段语义或允许跳过证据门禁。
- 非目标：由 Saber 运行 MCP server、代理 MCP tool call 或提供 MCP runtime。
- 非目标：改变业务仓独立 Git 边界和 L2/L3 安全策略。

## 方案与影响

| 领域 | 提议变更 | 风险 | 验证方式 |
| --- | --- | --- | --- |
| 初始化 | `saber init --tool <tool>` 安装统一能力 | 初始化失败可恢复 | CLI 测试与真实初始化验收 |
| 物化 | 一次投影全部阶段资产，不再选择角色 | 资产冲突 | 三工具物化测试 |
| 工作项 | 阶段转换记录在状态历史，不强制 handoff 文件 | 历史可读性 | 工作项迁移与状态测试 |
| MCP | 直接写 AI 工具原生 MCP 配置 | 工具格式差异 | 三适配器安装/卸载测试 |
| 路由 | `/saber` 根据对话意图和当前状态选择阶段 | 误路由 | 技能契约测试与门禁测试 |

## 验收条件

1. 初始化和物化命令不要求 BA、Dev 或 QA 角色参数。
2. 新成员选择 Codex、Claude Code 或 OpenCode 后可获得统一 `/saber` 命令和全部阶段工作流。
3. 阶段转换不创建或要求 `handoffs/*.md`，但状态历史和必要证据仍可复查。
4. MCP 安装直接使用配置声明，不启动 Saber bridge/runtime；卸载仅删除 Saber 管理的配置。
5. 历史工作项和兼容读取逻辑全部删除；新工作项只使用统一阶段历史模型，业务仓源码不进入 Saber 仓库。
6. `npm test`、`npm run check`、`npm run build` 和 `git diff --check` 通过。

## 门禁与批准

- 进入 Apply 的条件：仓库维护者已确认上述理想使用方式。
- L2 外部写入：无；本次只修改本地仓库。
- 暂停条件：需要恢复旧工作项兼容，或改变外部写入安全边界。
- 批准记录：2026-07-23，维护者确认团队成员不区分固定角色，由对话内容灵活进入各阶段。
- 范围变更批准：2026-07-23，维护者明确允许删除历史工作项，不保留兼容。
